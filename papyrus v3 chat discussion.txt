🤖 Papi — The Meta-Bot That Watches Papyrus
The Core Problem You're Solving

Nobody is "watching the watcher." Papi fills that gap.

What Papi Would Actually Do (Practical)
| Layer | What Papi Monitors | Action |
|-------|-------------------|--------|
| **Health** | Memory usage, CPU, process stability on i5/8GB | Warn before crash, suggest restart |
| **LLM Connection** | API timeouts, 429s, 400s, token limits | Auto-switch provider, notify user |

Is It Too Much for i5 6th Gen / 8GB RAM?
No. Here's why:

Papi is not a heavy local model. It's a lightweight watchdog — essentially a Node.js event loop with timers and hooks. The actual intelligence (deciding what to do) can optionally call the same OpenRouter API Papyrus uses, but most of Papi's work is:
✅ Lightweight: Event listeners, timers, counters (negligible RAM)
✅ Lightweight: Logging to a local SQLite file (~5MB)
✅ Lightweight: Process health checks every 30s (near-zero CPU)
⚠️ Optional: Quick LLM call for error diagnosis (only when something breaks)
Estimated overhead: ~20-40MB RAM, <1% CPU. Your 8GB machine won't even notice it.

| Operation | Node.js | Bun | Speedup |

Impact: Papyrus starts in a blink, file tools feel instant, and you skip the entire TypeScript compilation step.

🟡 Go — Only for One Specific Thing
Go is blazing fast for compiled binaries, but rewriting Papyrus in Go means rewriting 2300+ files. Not worth it.

BUT — Go is perfect for one thing: Papi's health monitor daemon.
If Papi Core (the watchdog) is a tiny Go binary:

  • Uses ~5MB RAM
  • Runs as background process
  • Checks Papyrus health every 30s
  • Near-zero CPU overhead
A Go daemon for Papi's health monitoring is optional but elegant. It's the difference between "Node.js process watching Node.js process" (overhead) vs "tiny Go binary watching Node.js process" (negligible).

Impact: Low for Papyrus itself, but high for making Papi feel invisible.

🟢 TypeScript — Already the Core
